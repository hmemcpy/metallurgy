Modifications made to copied source
-----------------------------------
(1) base/package.scala
    Changed `import junit.framework.TestCase` to
    `import _root_.junit.framework.TestCase`. The bundled Scala plugin jar
    `scalaCommunity.junit.jar` ships an (almost empty)
    `org.jetbrains.plugins.scala.junit` package which shadows the bare `junit`
    import under Scala's relative-package resolution. The `_root_` qualifier
    forces absolute resolution to JUnit's `junit.framework.TestCase`. This is
    the standard fix for this Scala import-shadowing gotcha.

(2) SourceRootTestUtil.scala and util/Annotations.scala
    Both were not in the task brief but are referenced by other copied files
    (ScalaLightCodeInsightFixtureTestCase.scala and
    MultipleScalaVersionsJUnit4Runner.scala respectively), so they had to be
    brought across as well. Both are unchanged from upstream.

(3) base/libraryLoaders/{LocalJarLibraryLoader,PlatformSdkJdkLoader,ScalaLibraryLoader}.scala
    Copied verbatim. Not in the original task brief, but the upstream
    libraryLoaders directory ships them alongside the explicitly-listed
    loaders; without them the package would not compile (LibraryLoader
    references them). All three are unchanged.

(4) util/runners/TestScalaVersion.java
    The `Scala_3_9` enum constant and its switch arm are commented out. The
    upstream idea261.x source references `LatestScalaVersions.Scala_3_9`, but
    that constant does not exist in the bundled Scala plugin version 2026.1.20
    (which only goes up to Scala_3_8). Re-enable both lines when the bundled
    Scala plugin is upgraded past 2026.1.20.

(5) util/dependencymanager/TestDependencyManagers.scala
    The task brief listed this file as "TestDependencyManager.scala"; the
    upstream file is actually named `TestDependencyManagers.scala` (plural).
    It is copied verbatim under its real upstream name.


Build configuration changes
---------------------------
In addition to the source files above, the following build configuration
was modified to make the backport compile against the IntelliJ 261.x
modular SDK:

  build.sbt
    - Added `-Ytasty-reader` to scalacOptions. Required because the bundled
      Scala plugin ships TASTy metadata (e.g. for `project.Version`) that
      scalac needs to read.
    - Added `intellijRepositoryReleases` resolver pointing at
      https://www.jetbrains.com/intellij-repository/releases
    - Added `intellijDependencies` resolver pointing at
      https://cache-redirector.jetbrains.com/intellij-dependencies
      (transitive deps of the test-framework artifacts).
    - Added `% Test` dependencies on the split IntelliJ test-framework
      Maven artifacts, because the modular 261.x SDK no longer ships
      `lib/testFramework.jar` (it has been removed from the IDE distribution
      and is now published only as separate Maven artifacts keyed by the
      platform build number 261.26222.65):
        com.jetbrains.intellij.platform:test-framework-core
        com.jetbrains.intellij.platform:test-framework-common
        com.jetbrains.intellij.platform:test-framework
        com.jetbrains.intellij.platform:test-framework-junit5
        com.jetbrains.intellij.java:    java-test-framework-shared
        com.jetbrains.intellij.java:    java-test-framework-backend
        com.jetbrains.intellij.java:    java-test-framework

This resolution pattern is the same one used by the upstream
JetBrains/intellij-scala project (see project/dependencies.scala lines
207-218) — they had to solve the same problem when their build moved past
the modular-platform break in 2024.2.
