# 15 — Where Scala 3 IDE support actually lacks today (type resolution, macros, new features)

A primary-source dive to size Metallurgy's refocused opportunity (ADR 0011): **type resolution where the
bundled plugin falls back to `Any`/wrong, plus macro-heavy and new-feature Scala 3.** The headline finding
from the dogfood inspection — only jing's `inlineYaml` resolves to `Any` — is the visible tip; this doc sizes
the iceberg beneath it.

## Sources & method

- **YouTrack `SCL`** (JetBrains Scala plugin tracker) — the authoritative source for bundled-plugin gaps.
  Per-term queries over `State: Open` filtered to Scala 3 (`#Scala 3` tag + text). Issue IDs + vote counts
  below are the evidence; subsystems noted where salient.
- **`scala/scala3` GitHub** — the compiler repo carries no IDE label and surfaces little IDE-specific
  triage (IDE bugs live in SCL); the presentation compiler (`InteractiveDriver`) is what Metallurgy *uses*,
  so its correctness is Metallurgy's floor, not its ceiling.
- **Scala Discourse** (`users.scala-lang.org`) — weak, generic signal ("scala-incorrect-type-inference-
  intellij", "scala-intellij-plugin-broken-with-3-5"); confirms pain exists but adds no specificity over SCL.

The dogfood inspection (25 samples, clean IntelliJ, ADR 0011) is the human ground-truth; this dive sizes
what the dogfood doesn't yet cover.

### D. Macro annotations (Scala 2 `scalameta` today; Scala 3 `MacroAnnotation` coming)

| Issue | Votes | Gap |
|---|---|---|

| SCL-9209 | 3 | Fields generated by a macro aren't highlighted (MacroParadise) |
| SCL-12463 / SCL-13323 | 1 | Good-code-red inside macro-annotated classes |

Scala 3's **new `scala.annotation.MacroAnnotation`** (experimental, the successor to scalameta/paradise
macro annotations — generates members/symbols at the annotation site) has **zero SCL issues filed**: it is
uncharted for the bundled plugin. Macro annotations generate *symbols the PSI never sees*, so resolve /
type / completion all fail until the annotation runs. This is a high-value, predictable gap.

### E. Caprese (capture checking) — closures & escape analysis: **uncharted**

`caprese` / `capability` queries return no capture-checking-specific issues (the `capture` hits are all
akka/play/spray noise). Caprese is too new/experimental to have demand signal in SCL. The construct space
the bundled plugin certainly does not model:
- **Closure capture / escape**: `def use(f: File^): String = f.read` — whether a captured `^` cap may flow
  out of a closure (`{ () => f }^`, `Cap → Cap2`).
- **Boxed caps** (`{1}^`), **`@capability`** params, **`try`/`with`** resource scopes, **reach** caps.
- **`caps`/`effect`** systems building on it.

No filed demand, but if/when Caprese stabilizes (Scala 3 roadmap), this becomes a green field. Metallurgy
could pioneer it; there is no user pressure yet.

### F. New-feature gaps (Scala 3.5+) — the leading edge

Where the language moves faster than the bundled plugin's hand-rolled typer:
- **Named Tuples** (3.5) — SCL-25016, SCL-24028 (type → `Any`, good-code-red).
- **Polymorphic function types** — SCL-23315 ("unexpected Polymorphic function literals must have a value
  parameter"), SCL-23267 (quick-doc), SCL-20759 (polymorphic eta-expansion, SIP-49).
- **Context functions** — SCL-24214 (completion of extension methods on context functions), SCL-22529.
- **Deferred givens** (`given ... = deferred`) — SCL-24051 (local given not recognized), SCL-24721.
- **`MacroAnnotation`** (above).
- **Export clauses / top-level exports** — SCL-23395, SCL-21758.
- **Opaque types** — SCL-19245 (debugger eval of opaque-type methods).

## Reconciliation with ADR 0010

ADR 0010 ("CBH resolves Scala 3 at steady state") holds for the **simple stdlib + circe** set the triages
measured. This dive refines it: the bundled plugin is **not** broadly correct — it lags on (a) type-level
computation (`compiletime.ops`, match types, named tuples), (b) macro/typeclass derivation (shapeless,
frameless, chisel, refined), (c) macro annotations (symbol generation), and (d) new 3.5+ features. Those are
real, voted, open gaps — exactly the `Any`/wrong-type/wrong-red category the dogfood's jing sample
represents.

## Where Metallurgy earns its place (refocused, prioritized)

Mapped to `pc`'s strengths (real `dotc` typer → expands macros, reduces type-level computation, resolves
derived/refined types):

1. **Type resolution for type-level computation** (A): `compiletime.ops` literals, match types, named
   tuples, polymorphic/context functions. Broad, voter-light but high-frequency. *Enrich* the
   `CompilerType` slot + hover/inlay.
2. **Macro/typeclass derivation libraries** (C): shapeless, frameless, chisel, refined, circe-derivation,
   magnolia. `pc` runs the real typer, so `summon`/derivation resolves. Niche but vocal users.
3. **Scala 3 `MacroAnnotation`** (D): symbol-generating annotations — a predictable, near-total gap (the
   PSI never sees generated symbols). High value if the feature lands.
4. **Inlay hints parity with Scala 2** (B): call-chains, local-variable types — `pc` has the types.
5. **New 3.5+ features** (F): named tuples, deferred givens, exports — keep pace with the language.
6. **Caprese closures/escape** (E): green field, no demand yet — defer until Caprese stabilizes.

The common thread: every one is a **resolve/type/completion** gap (the *Enrich* pattern from CONTEXT), not a
red-suppression gap. The #3 highlighting pipeline (ADR 0009) is not the tool for these — `pc`-driven
**type resolution + completion** is.
